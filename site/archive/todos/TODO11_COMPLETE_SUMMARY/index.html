    Join Strategies (3)¶

    1. INTERSECTION (AND Logic)
    2. Returns documents present in ALL databases
    3. Use case: High-confidence document matching

    4. Example: Vector AND Graph AND Relational

    5. UNION (OR Logic)

    6. Returns documents present in ANY database
    7. Use case: Comprehensive document discovery

    8. Example: Vector OR Graph OR Relational

    9. SEQUENTIAL (Pipeline)

    10. Uses results from DB1 to filter DB2, then DB3, etc.
    11. Use case: Progressive refinement of search results
    12. Example: Vector â†’ Graph â†’ Relational (pipeline)

    Execution Modes (3)¶

    1. PARALLEL
    2. All database queries execute simultaneously
    3. Uses ThreadPoolExecutor

    4. Best for INTERSECTION and UNION

    5. SEQUENTIAL

    6. Queries execute one after another
    7. Allows progressive filtering

    8. Required for SEQUENTIAL join strategy

    9. SMART (Recommended)

    10. Automatically chooses best mode
    11. PARALLEL for INTERSECTION/UNION
    12. SEQUENTIAL for SEQUENTIAL join

    Query Builders (4)¶

    1. VectorQueryBuilder - by_similarity(embedding, threshold, top_k)
    2. GraphQueryBuilder - by_relationship(type, direction, max_depth)
    3. RelationalQueryBuilder - from_table().where().limit()
    4. FileStorageQueryBuilder - by_extension(extensions, directory)

    Query Execution Flow¶

    1. Create PolyglotQuery(unified_strategy)
2. Configure databases via query builders
   - query.vector().by_similarity(...)
   - query.graph().by_relationship(...)
   - query.relational().from_table(...).where(...).limit(...)
3. Set join strategy
   - query.join_strategy(JoinStrategy.INTERSECTION)
4. Execute
   - query.execute()
5. Determine execution mode (SMART logic)
6. Execute queries (PARALLEL or SEQUENTIAL)
7. Join results based on strategy
8. Return PolyglotQueryResult
